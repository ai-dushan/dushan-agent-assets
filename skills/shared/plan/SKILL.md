# 实施计划编写与执行跟踪 (Agentic Plan)

> **用法**：当用户下发复杂的编码任务，或显式调用 `/plan` 时触发此 Skill。
> **核心目标**：提供结构化的任务拆解、风险评估与执行跟踪。确保需求无遗漏、架构变更可控、执行过程完全透明。

## 〇、核心执行规范

1. **前置代码分析**：禁止凭空捏造文件路径或 API。在制定计划前，必须通过工具（如 `view_file`, `grep_search` 或 MCP 工具）扫描并理解相关代码上下文。
2. **需求对齐**：如果用户需求存在逻辑缺失或边界不清，必须先通过对话提问澄清，确认无误后再生成实施计划。
3. **架构纪律**：评估旧代码的保留价值，若需废弃应整体重构并修改调用方，避免堆砌向后兼容的临时代码。
4. **防御性编程限制**：避免使用 `dict.get()`、`getattr(x, k, default)` 等掩盖潜在异常的防御性写法，遵循快速失败原则。
5. **极度细致的留痕**：每改动一行代码或完成一个步骤，必须在计划文档中详细记录修改的目录、文件、行数及目的，并打上时间戳 `✅ 完成 HH:MM`，禁止事后补记。

---

## Step 1: 需求澄清与上下文收集

> ⚠️ **注意**：接到任务后，必须先完成此步骤，不可跳过直接生成计划文本。

1. **需求解构**：阅读用户描述，提取核心业务目标和技术难点。
2. **主动发问**：针对以下潜在问题向用户澄清：
   - 数据流或状态流转是否存在闭环问题？
   - 是否涉及核心数据库表结构变更？
   - 是否存在并发或安全性边界问题？
3. **全局勘探**：遍历系统当前状态：
   - 寻找现有类似功能的参考实现。
   - 确定受影响的依赖模块（评估修改范围）。
   - 查阅知识库 (`KI`) 或 `.md/` 目录下的历史设计文档。

---

## Step 2: 建立计划文档

在项目的 `.md` 目录下创建 Markdown 跟踪文件：

**文件路径规范**：

```text
{项目根目录}/.md/{YYYYMMDD}/{功能简写中文}-plan.md
```

_(例: `.md/20260418/全局异常拦截器重构-plan.md`)_

---

## Step 3: 计划文档模板

在上述文件中，严格采用以下结构并根据勘探结果填充：

```markdown

> **状态**：⬜ 计划中 / 🔄 执行中 / ✅ 已完成
> **预计完成**：YYYY-MM-DD HH:MM
> **实际完成**：—
> **ROI 评估**：传统人工预计 {X} 小时 → AI 辅助预计 {Y} 小时 (效率提升 {Z} 倍)

- **影响范围**：预估会牵连修改的核心模块。
```

## Step 4: 标准执行流

执行计划时，遵循"编码-校验-记录"的标准化流程：

1. **精确编码**：按照计划步骤，使用 `view_file` 等文件修改工具进行代码编写。
2. **遵循规范**：严格遵守当前项目的 `RULES` (如 `.agents/rules/global.md`) 或 MCP 规范知识库。
3. **质量校验**：每次文件修改完成后，必须调用 Lint 工具（如 `forge-lint`，或 `npm run lint`）。确保零 Error 方可进入下一步。
4. **精细化留痕**：实时更新 Plan 文档：
   - 将 `- [ ]` 改为 `- [x]`，追加时间戳 `✅ 完成 10:30`。
   - **填写执行记录**：在步骤下方明确列出实际修改的**目录**、**文件名**、**具体行数**（如 Lines 10-25），以及本次修改的**目的**。
   - **完成交叉审查标记**：添加 `👁️ 交叉审查：✅ 已验证` 标记。
   - 同步更新底部的《交叉验证清单》表格。
5. **记录修正**：若实际开发发现需增加额外操作，在原计划下追加 `⚠️ 修正：...` 并说明原因。

---

## Step 5: 收尾与复盘

所有开发阶段完成后，进行最终验收：

1. **全局测试**：运行项目相关的测试脚本以验证功能完整性。
2. **状态更新**：将文档头部状态更新为 `✅ 已完成`，并记录 `实际完成` 时间。
3. **补充复盘**：在文档末尾追加总结：

```markdown
## 五、复盘与下一步建议

- **实际耗时**：{总耗时}
- **AI 效能 ROI 复盘**：{实际人工干预耗时} vs {纯人工开发预估}，计算真实提效倍数。
- **偏差反思**：{实际开发与预估的差异原因}
- **架构沉淀**：{本次改动产生的新设计模式或规范建议}
- **下一步行动**：
  1. {行动 1}
  2. {行动 2}
```

4. **进度汇报**：向用户输出精炼的总结报告，并列出建议的后续指令。

---

## 开发避坑指南

- ❌ **直接输出长篇代码片段**：应优先使用后台文件操作工具完成修改，并通过更新 plan 汇报进度，保持聊天界面清爽。
- ❌ **留痕不细致**：严禁只打个勾。**必须**写清楚修改了哪个目录、文件、哪些行，以及为什么修改，方便后续问题回溯与交叉审查。
- ❌ **延后更新进度**：严禁多个步骤完成后统一打勾。每完成一步必须立即更新文档状态，确保追踪准确性。
